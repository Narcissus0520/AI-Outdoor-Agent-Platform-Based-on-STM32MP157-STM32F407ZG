# AGENTS.md

本文件用于指导 Codex 在本仓库中高效、稳定地工作。

本项目是一个长期嵌入式作品集项目，目标是围绕 `STM32MP157 + STM32F407ZG` 构建一个户外智能体平台，逐步体现嵌入式 Linux、C/C++ 工程化、GNSS、Sensor Integration、IPC、Runtime Service 和 AI Terminal 相关能力。

---

## 1. 项目定位

项目名称：

```text
AI Outdoor Agent Platform
```

项目方向：

```text
基于 STM32MP157 + STM32F407ZG 的户外智能体 / AI Outdoor Agent Platform
```

本项目不是简单 Demo，而是一个面向嵌入式软件工程师跳槽作品集的长期项目。

项目需要体现：

* 嵌入式 Linux 用户态开发能力
* C / C++ 工程化能力
* Linux Runtime Service 设计能力
* GNSS 定位数据处理能力
* Sensor 数据抽象与集成能力
* STM32MP157 与 STM32F407ZG 异构协作能力
* IPC、日志、配置、进程管理等系统编程能力
* 面向 AI 终端系统的长期演进能力

## 5. 代码修改原则

### 基本原则

* 保持代码简单、清晰、可维护。
* 优先使用明确的模块边界。
* 避免一次性大规模重构。
* 不要为了炫技引入复杂设计模式。
* 不要写与当前阶段无关的大量占位代码。
* 不要删除已有有效代码，除非任务明确要求或有充分理由。
* 如果修改了公共接口，需要同步更新调用方和文档。

### C / C++ 代码要求

* C++ 代码优先使用现代 C++ 风格，但不要过度模板化。
* 优先使用 RAII 管理资源。
* 避免裸指针所有权不清。
* 注意嵌入式环境中的内存、CPU 和依赖成本。
* 尽量避免异常作为主流程控制。
* 日志、错误码、返回值要清晰。
* 对外接口命名应稳定、直观。

### 命名建议

文件和目录命名优先使用小写加下划线：

```text
gnss_service.cpp
runtime_manager.cpp
config_loader.cpp
ipc_channel.cpp
```

类名使用大驼峰：

```cpp
GnssService
RuntimeManager
ConfigLoader
IpcChannel
```

函数名和变量名使用小驼峰或下划线风格，但同一模块内要保持一致。

## 18. 问题排查记录约束

本项目的问题排查过程是作品集和面试资料的重要组成部分。后续凡是遇到需要分析的构建失败、测试失败、运行异常、硬件联调问题、性能退化或数据异常，都必须同步维护：

```text
docs/troubleshooting_log.md
```

记录要求：

* 问题开始分析时就创建唯一问题 ID，不要等修复完成后只补最终结论。
* 必须记录问题现象、影响范围、软硬件环境、接线与供电前提、复现步骤、预期结果和实际结果。
* 必须记录排查假设、依据、尝试动作和实际结果；无效尝试、回退方案和误判也必须保留。
* 根因必须有日志、错误码、状态位、抓包、测量、测试或代码路径作为证据；不能确认时明确写“根因未完全确认”。
* 修复必须记录修改文件、方案取舍和潜在副作用。
* 验证必须记录实际执行的命令、时长、次数、关键指标和结果；未执行的验证不能写成通过。
* 问题关闭后若再次出现，应追加证据并重新打开或建立关联问题，不能覆盖原有历史。
* 重要技术取舍关联 `docs/adr/`，阶段性实现关联 `docs/dev_log.md`，必要时在 README 或设计文档中同步当前真实状态。
* 每条问题记录最后提炼面试讲述要点，使用“现象 -> 缩小故障域 -> 关键证据 -> 方案取舍 -> 验证 -> 剩余风险”的结构，禁止夸大未验证结论。

问题修复任务完成前，Codex 应检查对应问题记录是否形成“现象、排查、根因、修复、验证”闭环。如果没有闭环，问题只能标记为“分析中”“已修复待验证”“已恢复，根因未完全确认”或“长期观察”，不能标记为“已关闭”。
