# ADR-0001: 使用 CMake 管理 Stage 0 C++ 工程

## 状态

Accepted

## 背景

Stage 0 需要建立一个可编译、可扩展的 C++17 Linux 用户态 Runtime 工程。后续项目会逐步扩展 Runtime Service、GNSS、Sensor Integration、IPC 和 STM32MP157 交叉编译支持，因此构建系统需要便于组织模块、管理 include 路径，并为后续交叉编译预留空间。

## 可选方案

### 方案 A: 手写 Makefile

优点：

- 简单直接，依赖少。
- 对小型 C/C++ 项目足够轻量。

缺点：

- 后续模块增多后维护成本上升。
- 交叉编译、测试和多目标管理需要手工补充较多规则。

### 方案 B: 使用 CMake

优点：

- 适合 C/C++ 工程化组织。
- 便于后续添加库、测试、安装规则和工具链文件。
- 对 STM32MP157 Embedded Linux 交叉编译支持更自然。

缺点：

- 比 Makefile 多一层配置抽象。
- 本地环境需要安装 CMake。

## 最终决策

Stage 0 使用 CMake 作为主构建系统，并保持当前配置最小化。

## 决策理由

CMake 更适合作为长期作品集项目的基础构建系统，可以支持后续模块化、测试和交叉编译，同时不会在 Stage 0.1 引入大型第三方依赖。当前 `CMakeLists.txt` 只定义 C++17、源码列表、include 路径和基础编译警告。

## 风险

- 如果开发环境未安装 CMake，无法直接执行标准构建命令。
- 后续交叉编译工具链文件需要单独维护，避免污染本机构建配置。
