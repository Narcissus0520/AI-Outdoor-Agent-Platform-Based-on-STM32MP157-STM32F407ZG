# ADR-0014: Recover I2C2 After Runtime Transfer Errors

## 状态

Accepted

## 背景

ICM42688、MMC5603 和 BMP390 共用 F407 I2C2。2026-07-19 的 30 秒持续抓包在约 27 秒后观察到三路真实数据同时停止，heartbeat 从 `0x0029` 变为 `0x0056`。原实现只在启动时释放总线；运行期 HAL I2C 事务失败后直接向上返回错误，无法清除 STM32F4 I2C 外设 BUSY 状态或从机持低 SDA 的情况。

## 可选方案

### 方案 A：仅保留错误状态，等待人工复位

优点：实现最简单，不增加错误路径代码。

缺点：一次瞬时总线错误就会让三颗传感器持续失联，不适合长期运行的 Sensor Hub。

### 方案 B：只重新调用 HAL I2C 初始化

优点：改动较小。

缺点：无法可靠释放被从机持低的 SDA，也不能保证清除 STM32F4 I2C 外设内部 BUSY 状态。

### 方案 C：硬复位外设、GPIO 释放总线并重试一次

优点：同时处理外设状态机卡死和未完成从机事务；只在首次失败时付出恢复开销，正常采样路径不增加额外事务。

缺点：恢复期间会短暂阻塞主循环；若线路断开、供电异常或从机永久损坏，单次重试仍会失败。

## 最终决策

选择方案 C。`board_i2c_mem_read()` 和 `board_i2c_mem_write()` 首次失败后调用 `MX_I2C2_Recover()`：先 HAL 去初始化并硬复位 I2C2，再将 PB10/PB11 切为开漏 GPIO，根据 SDA 状态发送最多 18 个 SCL 脉冲，明确生成 STOP，恢复 I2C2 复用功能后重试原事务一次。

## 决策理由

三颗器件同时失联说明故障域位于共享总线或控制器。恢复动作放在 BSP 层可覆盖所有 provider，并保持传感器驱动不依赖 STM32 HAL。单次重试避免永久硬件故障时无限阻塞，同时保留现有 heartbeat error/fallback 状态作为上层故障指示。

## 风险

- 每次失败最多增加一次总线释放和一次事务超时，短时会降低帧率。
- GPIO 恢复无法修复断线、错误电平、缺少上拉或传感器供电异常。
- 当前 heartbeat 未暴露恢复计数，无法从 MP157 直接区分“从未失败”和“已自动恢复”。
