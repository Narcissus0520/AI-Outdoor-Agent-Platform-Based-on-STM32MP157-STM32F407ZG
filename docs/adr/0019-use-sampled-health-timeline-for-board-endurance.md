# ADR-0019: 板端耐久验收使用逐秒健康时间线

## 状态

Accepted

## 背景

板端小时长测原先只在结束时读取 `runtime_status.json`。2026-07-19 的一次正式长测在启动预检为 `0x01A9` 后，运行约两分钟出现 `0x022E` 和 `0x03AE`，表示 ICM42688 FIFO error/fallback，随后又自动恢复为 `0x01A9`。如果只检查最终快照，这段真实降级会完全丢失。

ICM42688 固件本身包含 FIFO flush、重新初始化和 Mock fallback 恢复路径，因此“任何瞬态状态都判永久失败”也不能区分受控自恢复与持续故障。

## 可选方案

### 方案 A：只检查最终状态

优点：

- 实现简单，审计成本低。

缺点：

- 无法发现运行中曾经出现的 fallback、FIFO error 或错误字段。
- 最终恢复可能掩盖真实的数据质量缺口。

### 方案 B：只要出现一次非 `0x01A9` 就失败

优点：

- 门槛最严格，规则容易解释。

缺点：

- 无法体现固件已设计的瞬态恢复能力。
- 1 秒采样边界附近的单次状态可能让整小时测试失去诊断价值。

### 方案 C：逐秒采样并限制降级比例和连续时长

优点：

- 同时保留时间、状态位和核心错误字段，可还原故障持续时间。
- 可区分短暂自恢复、持续 fallback 和不可恢复的致命状态位。
- 结果可由脚本重复审计，适合长期测试和面试复盘。

缺点：

- 1 秒粒度无法捕获更短的瞬态。
- 需要维护采集进程、完成标记和验收阈值。

## 最终决策

采用方案 C。`monitor_board_runtime_health.sh` 每秒记录 Runtime state、F407 `status_flags`、Runtime/storage/MCU/Board IMU 四个 error，以及 Sensor Hub I2C/FIFO 累计诊断。长测审计要求：

- 时间线样本数不少于计划秒数减 5；初始状态缺失不超过 3 个样本。
- bit 10 至 bit 14 的 TX overflow、初始化失败、RX overflow、I2C 最终失败等致命位从未出现。
- 四个核心 error 在所有有效样本中均为空。
- `0x01A9` 健康样本占有效样本至少 99%。
- 连续非 `0x01A9` 不超过 5 个逐秒样本。
- 最终状态必须恢复为 `0x01A9`。
- diagnostics 必须被 MP157 解码；累计计数不得回退，测试期间 I2C final failure、FIFO overflow/malformed/stall 不得增加。

健康时间线是最终状态、CSV 完整性/频率和日志轮转之外的附加门槛，不能替代其他审计。

## 决策理由

该方案把“是否恢复”和“恢复用了多久”都变成证据。99% 与 5 秒门槛允许验证现有恢复机制，同时会拒绝持续 fallback、频繁抖动或任何致命状态位。实现只依赖 POSIX shell 和 AWK，不增加 Runtime 或交叉编译依赖。

## 风险

- 1 秒采样可能漏掉亚秒级错误；当前只能结合 CSV 频率和最终状态间接覆盖。
- 99% 和 5 秒是 Stage 1 初始验收阈值，获得更多小时级样本后可能需要收紧。
- diagnostics 与高频业务帧共享 UART4 队列；需要继续通过 TX overflow sticky bit 和完整时间线确认诊断流量没有反向影响业务链路。
