# ADR-0022: 在 MP157 Runtime 计算可标定的倾斜补偿罗盘航向

## 状态

Accepted；算法、配置和主机集成验证已完成，真实板安装矩阵与标定系数仍待采集。

## 背景

F407 已通过 MMC5603 输出 20 Hz 三轴磁场，通过 ICM42688 输出约 100 Hz 加速度/陀螺仪，但 MP157 只发布原始数据。framebuffer 方向卡片此前在没有 GNSS course 时使用固定 `62.3°` 展示值，不能作为真实航向证据。

罗盘解算需要同时处理：

- 硬铁偏置；
- 软铁畸变和传感器相对机身的轴向安装关系；
- 由加速度估计的横滚、俯仰和倾斜补偿；
- 磁偏角；
- 动态加速度、异常磁场和不同步样本的质量门槛。

当前还没有围绕整机完成多方向旋转采样，因此不能把默认系数描述为真实标定结果。

## 可选方案

### 方案 A：在 F407 固件中计算航向并增加协议字段

优点：输出靠近传感器，时间同步直接。

缺点：需要修改 MCU 协议和固件；浮点/矩阵配置、现场系数更新和诊断在 MCU 侧维护成本更高，也会把板级安装参数固化进 Sensor Hub。

### 方案 B：在 MP157 Runtime 中对现有 IMU/磁场帧进行解算

优点：不改变已验证协议；C++ 可清晰实现矩阵、质量状态和配置校验；现场更新参数无需重新烧写 F407；结果可直接进入 JSON 和 dashboard。

缺点：两类帧异步到达，需要样本时间差门槛；Linux 侧只看到 F407 已发布的数据。

### 方案 C：引入完整 AHRS/姿态融合第三方库

优点：可进一步融合陀螺仪，动态姿态更稳定。

缺点：当前只需完成罗盘基础链路；第三方算法会增加交叉编译、参数调试和维护成本，且无法替代真实磁场标定。

## 最终决策

采用方案 B，暂不引入第三方依赖：

- 机身坐标约定为 `+X` 向前、`+Y` 向右、`+Z` 向上；完整 3×3 矩阵同时承载软铁修正和传感器到机身的安装变换。
- 先减去三轴硬铁偏置，再乘 3×3 矩阵。
- 使用加速度计算 roll/pitch，对磁场执行倾斜补偿；磁航向归一化到 `[0, 360)`，再叠加可配置磁偏角。
- IMU/磁力计时间差默认不超过 250 ms；加速度模长默认限定为 `0.70–1.30 g`，磁场模长默认限定为 `5–100 uT`。
- 只有 heartbeat 同时确认真实 ICM42688 ready、MMC5603 ready，且 fallback、IMU/MMC/FIFO/init/I2C 当前错误位均未置位时才接受输入；`0x102E` 等 Mock fallback 状态不能产生有效航向。
- 系数矩阵必须有限且非奇异，范围上下限必须有效；无效配置拒绝启动。
- 默认使用单位矩阵和零偏置，但 `compass_calibration_valid=false`。此时仍可输出 `quality=uncalibrated` 的原始估计，不能写成已标定航向。
- `runtime_status.json` 增加独立 `compass` 节点；text/framebuffer dashboard 使用有效罗盘航向，罗盘无效时只在有效 GNSS fix 且速度不低于 1 km/h 时回退 GNSS course，不再使用固定假航向。

## 决策理由

该方案最大限度复用已有数据链路，把设备安装和地理位置相关参数留在 Linux 配置层，并使“算法可运行”和“真实标定完成”成为两个可单独验证的状态。固定资源的 F407 继续专注采集和确定性协议，MP157 承担更适合配置化和浮点计算的派生导航状态。

## 风险

- 加速度只能在近似静态时表示重力；剧烈运动时当前实现会因模长门槛拒绝输出，而不是假装姿态可信。
- 默认轴向约定尚未通过整机旋转验证；实际模块方向必须通过 3×3 矩阵修正。
- 室内电机、扬声器、钢结构和供电电流可能污染磁场；仅通过算法测试不能证明现场精度。
- 当前没有融合陀螺仪，动态航向稳定性低于完整 AHRS。
